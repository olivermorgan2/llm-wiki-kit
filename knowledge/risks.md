# Risk register — llm-wiki-kit

Live risks with mitigation and status. Seeded from PRD §16
([`design/prd.md`](../design/prd.md)) plus cross-cutting risks called out
elsewhere in the PRD. Status: `open` (mitigation planned, not yet built),
`mitigated-by-design` (addressed in PRD/architecture), or `closed`.

## Product / design risks (PRD §16)

| Risk | Mitigation | Status |
|---|---|---|
| Profile rules mistaken for OKF rules | Report OKF and profile conformance separately; never present a profile rule as universal OKF. Bounded by [ADR-004](../design/adr/adr-004-validation-and-severity-model.md) (accepted). **Exercised Phase 4:** the shipped `academic-research` profile's rules are all tagged `ruleset: profile` (the `profile-*` code family, ADR-010) and reported separately from OKF; the fixture/unit tests assert the split (criterion 5 holds for a real domain profile, not just core). Risk stays **open**, narrowed to cross-surface parity (Phase 6). | open |
| Schema becomes rigid too early | Keep `core` conservative; prefer recommendations before requirements. Bounded by [ADR-010](../design/adr/adr-010-profile-data-schema-and-rule-vocabulary.md) (accepted 2026-07-04): the profile schema fixes only the **closed vocabulary** addendum 003 needs; core rules stay engine-code and byte-identical (golden-parity constraint), and a fully-declarative core migration is a separate golden-guarded future issue rather than a rushed Phase-4 rewrite. | open |
| Hooks treated as a security boundary | Position CI as authoritative; document hook limitations; post-write hooks are feedback, not prevention. | open |
| Model overwrites human work | Preview existing-page changes; preserve unknown/unrelated content. Validation applies to human edits too — bounded by [ADR-004](../design/adr/adr-004-validation-and-severity-model.md) (accepted); safe-write gate [ADR-005](../design/adr/adr-005-safe-filesystem-layer.md) (accepted). | open |
| Sourced claims lose provenance | Require resolvable citations; preserve existing citations. Decided in [ADR-008](../design/adr/adr-008-provenance-and-citation-model.md) (accepted 2026-07-03, Codex `READY` 5/5) and **implemented + fixture-proven Phase 3**: the citation mechanism — evidence contexts, total deterministic offline three-class resolver, `core-citation-*` findings (#36/PR #46) — plus the plan-time citation-loss gate routed through the ADR-003 `approval` member (#37/PR #47); **criterion 9** now gates green (`TestAcceptanceCriterion9SourcedClaimCitationsResolveOffline`, #39/PR #49, with a non-vacuous negative control). Risk stays **open**, now narrowed past the Phase-4 profile-structural half, which is **implemented and fixture/acceptance-proven** (2026-07-04): the profile citation vocabulary and ADR-008's two carry-ins shipped as `profile-citation-required`/`profile-citation-target-type` (#56/`9497179`), fixture-proven by `internal/validate/academic_fixtures_test.go` (#57) and acceptance-proven by `TestAcceptanceCriterion4AcademicResearchEvidenceObligationJourney` on all five platforms (#59). The carry-ins are settled by [ADR-010](../design/adr/adr-010-profile-data-schema-and-rule-vocabulary.md): repo-path class 3 gated on `isIntraWiki`; a present-but-unresolved citation *satisfies* the require-citation obligation while `core-citation-unresolved` stays a separate promotable axis. Remaining scope: cross-surface (skills/hooks/CI/CLI) parity (Phase 6). Content-provenance only; supply-chain signing/attestation is a **separate deferred ADR** (see the supply-chain row below). | open |
| Custom-profile complexity becomes a programming language | Declarative data only; one inheritance level; explicit validation. Bounded by [ADR-007](../design/adr/adr-007-profile-system-boundary.md) (accepted) and now the concrete schema in [ADR-010](../design/adr/adr-010-profile-data-schema-and-rule-vocabulary.md) (accepted 2026-07-04): the profile vocabulary is a **closed key set** with the engine owning the rule *kinds* and the profile supplying only data; there is **no expression or conditional-section syntax** (the single `citation.requireWhen` field-equals-value trigger is the deliberately-weakest conditional). Any richer need takes a new addendum + ADR, not free-form authoring. | open |
| Cross-platform binaries drift | Automate release builds, checksums, and platform integration tests. Bounded by [ADR-002](../design/adr/adr-002-platform-binary-selection.md) (accepted). **Implemented** Phase 2: multi-platform release bundle + checksum manifest + per-platform selfcheck smoke (#17/PR #28) and a five-platform CI matrix (#15/PR #26). The **macos-amd64** CI leg formerly never dispatched (the retired `macos-13` label mapped to no runner); observation is **restored** by moving it to GitHub's last hosted Intel macOS image, `macos-15-intel` ([#30](https://github.com/olivermorgan2/llm-wiki-kit/issues/30)/PR #51), so amd64-macOS is again covered by observed CI. Residual: `macos-15-intel` is supported only through **Aug 2027**, after which hosted x86_64 macOS ends and amd64-macOS evidence needs a new decision (Rosetta-on-arm64 or a documented ADR-002 gate revision). windows-amd64 full suite is red on pre-existing `internal/` permission tests (follow-up [#29](https://github.com/olivermorgan2/llm-wiki-kit/issues/29)). | open |
| Compilation drops important research facts | Track evidence gaps; keep sources addressable; deeper evaluation later. | open |
| Upgrades overwrite customization | Track ownership; preview repository changes; preserve local profile extensions. Ownership model (plugin-owned vs repo-owned + version record) decided in [ADR-009](../design/adr/adr-009-install-upgrade-uninstall-ownership.md) (accepted). **Install half implemented** Phase 2: `.llm-wiki/manifest.json` version record + silent-overwrite refusal + non-empty-repo no-file-loss (#19/PR #25). Upgrade/uninstall preservation still Phase 7. | open |
| Skills bypass deterministic safety | Route managed mutations through staged engine plans; narrow tool access. Mandatory engine write-gate in [ADR-005](../design/adr/adr-005-safe-filesystem-layer.md) (accepted); staged inspect/plan/apply model in [ADR-006](../design/adr/adr-006-staged-mutation-transaction-model.md) (accepted). ADR-006 cross-file transaction layer implemented Phase 2 (#16/PR #23); the `inspect/plan/apply` UX half **implemented Phase 3** (`page plan` #34/PR #44, `page apply` #35/PR #45), and the authoring skill adapter (#38/PR #48) is a **thin adapter that routes all managed mutation through the staged engine plan** rather than writing directly — **criterion 12** proves writes land only via staged plan/apply (`TestAcceptanceCriterion12WritesOnlyThroughStagedPlanApply`, #39/PR #49). Risk stays **open**, narrowed to **cross-surface (skills/hooks/CI/CLI) parity**, which is Phase 6. | open |
| Skill adapters duplicate core logic | All policy/mutation in shared engine; test adapters as integrations only. | open |
| A reviewed diff becomes stale before application | Bind plans to source/target hashes; reject stale plans. Referenced by [ADR-003](../design/adr/adr-003-json-contract-and-exit-codes.md) (accepted); hash-binding mechanism owned by [ADR-006](../design/adr/adr-006-staged-mutation-transaction-model.md) (accepted). ADR-006 cross-file transaction commit — staging manifest with source/target base hashes + validate-before-commit — implemented Phase 2 (#16/PR #23); the `page plan`/`page apply` stale-plan **rejection UX implemented Phase 3** (#34/PR #44, #35/PR #45) and **acceptance-proven**: **criterion 13** shows a stale plan rejected with exit 4 / `invalid-invocation` and a **whole-tree byte-identical (zero-mutation)** snapshot (`TestAcceptanceCriterion13StalePlanRejectedZeroMutation`, #39/PR #49). Risk stays **open** with only a narrow residual (cross-surface parity, Phase 6) — the page-mutation path is now implemented and fixture-proven. | open |
| YAML round-trip drops unknown fields (comments best-effort) | Acceptance criterion 6 (binding) requires unknown **frontmatter field** preservation; comment preservation is a best-effort, non-gated design-quality bar. [ADR-001](../design/adr/adr-001-go-toolchain-and-yaml.md) (accepted) adopts a node-aware YAML library (`goccy/go-yaml`) over archived `yaml.v3`, preserving unknown fields (gated) and comments (best-effort); the unknown-field round-trip fixture **now gates green in Phase 3** — **criterion 6** proves unknown fields (and the edited known field) survive plan→apply in the committed live file, both directions (`TestAcceptanceCriterion6UnknownFieldsRoundTripPlanApply`, #39/PR #49). Risk stays **open** for the non-gated comment-preservation best-effort bar. | open |

## Cross-cutting risks (from elsewhere in the PRD)

| Risk | Mitigation | Status |
|---|---|---|
| Supply chain / tampered binary | Versioned, checksum-verified release artifacts; verify checksums before executing bundled CLI (PRD §9, §14 Security). Checksum-verify-before-exec is a decided chokepoint in [ADR-002](../design/adr/adr-002-platform-binary-selection.md) (accepted). **Residual risk:** the checksum manifest ships inside the same payload, so verification catches wrong-platform/mismatched/corrupt binaries but **not** a maliciously rebuilt payload (attacker can recompute the manifest). The trust root that would close this — release **signing / provenance attestation** — is explicitly **deferred** by ADR-002; [ADR-009](../design/adr/adr-009-install-upgrade-uninstall-ownership.md) (accepted) confirms signing/provenance is **out of its scope** and re-defers it to a dedicated supply-chain ADR; risk stays open. | open |
| Path traversal / symlink escape | Engine canonicalizes & validates paths; resolves symlinks; rejects escapes; bounded write scope (FR10, §14). Single mandatory write-gate in [ADR-005](../design/adr/adr-005-safe-filesystem-layer.md) (accepted) — criterion 17 hard gate. | open |
| Untrusted source material as instructions | Treat imported content as data, never as agent instructions; repo content cannot override plugin/user instructions (FR10, §14 Security). Inputs-as-data at the filesystem boundary **contributes to** this mitigation via the write-gate in [ADR-005](../design/adr/adr-005-safe-filesystem-layer.md) (accepted); the full instruction-injection boundary (skill prompting, adapter/model instruction handling) is not owned by ADR-005 alone. | open |
| Stale mutation plan applied after target changed | `apply` rechecks hashes and refuses stale/modified plans without mutating (FR7, FR11). Referenced by [ADR-003](../design/adr/adr-003-json-contract-and-exit-codes.md) (accepted); mechanism owned by [ADR-006](../design/adr/adr-006-staged-mutation-transaction-model.md) (accepted). ADR-006 base-hash validate-before-commit **implemented** Phase 2 (#16/PR #23); the `apply`-time stale-plan **rejection UX** is Phase 3. | open |
| Partial multi-file writes on interruption | Atomic writes; no partial updates across multi-file ops; safe recovery from interruption (FR1, FR10, §14 Reliability). **Per-file** atomic write+rename and the `.llm-wiki/` staging area are provided by [ADR-005](../design/adr/adr-005-safe-filesystem-layer.md) (accepted); the **cross-file** all-or-nothing transaction model (staging manifest, commit ordering, recovery/rollback, partial-commit detection) that makes a multi-file batch atomic is owned by [ADR-006](../design/adr/adr-006-staged-mutation-transaction-model.md) (accepted) and is **implemented** Phase 2 (#16/PR #23, `internal/txn`). ADR-005 alone gives the necessary primitive; ADR-006 completes full multi-file interruption safety. | open |
| Divergent findings across skills/hooks/CI/CLI | One deterministic implementation shared by all four; acceptance criterion requires identical findings for identical state (§17). Shared JSON envelope [ADR-003](../design/adr/adr-003-json-contract-and-exit-codes.md) + single validation engine [ADR-004](../design/adr/adr-004-validation-and-severity-model.md) (both proposed) — criterion 15. | open |
