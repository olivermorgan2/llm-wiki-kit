1. Verdict: NEEDS_REVISION

ADR-001 through ADR-005 are directionally coherent and mostly aligned with the MVP/build-out plan, but I would not accept them as-is. There are a few decision-quality gaps where the ADRs over-claim acceptance criteria, leave security/trust boundaries underspecified, or assert reliability guarantees without enough architectural mechanism.

2. Blocking findings

Blocking finding 1 — High
Severity: High

Exact evidence:
- ADR-001 Context: “Acceptance criterion 6 (addendum 001) requires unknown frontmatter fields and comments to survive authoring and enrichment — i.e. YAML round-trip preservation.”
- Addendum 001 criterion 6: “Unknown frontmatter fields survive authoring and enrichment.”
- MVP principle 5: “Unknown frontmatter fields and unrelated human content are never silently overwritten.”
- Build-out plan assumption: “acceptance criterion 6 requires unknown-field (and comment) round-trip preservation”

Why it matters:
ADR-001 makes `goccy/go-yaml` look mandatory partly because comments are allegedly part of the binding acceptance criterion. In the provided canonical acceptance criteria, comments are not actually named. The decision may still be correct, but the rationale currently inflates the ship gate and can accidentally create hidden MVP scope: comment preservation becomes treated as required even though the binding criterion only says unknown frontmatter fields survive.

Required fix:
Revise ADR-001 and related knowledge/build-out language to distinguish:
- binding acceptance requirement: unknown frontmatter fields survive;
- chosen design quality bar, if desired: preserve comments where feasible;
- explicit acceptance test boundary: what must fail the gate vs what is best-effort or non-blocking.

If comment preservation is intended to be binding, add it explicitly to an authoritative addendum or acceptance fixture rather than smuggling it through ADR rationale.

Blocking finding 2 — High
Severity: High

Exact evidence:
- ADR-002 Decision: “ship per-platform binaries in the plugin and select via a single OS+arch detection path that verifies the artifact checksum before execution.”
- ADR-002 Option A: “checksums shipped alongside are verified before execution”
- Risks: “Supply chain / tampered binary | Versioned, checksum-verified release artifacts; verify checksums before executing bundled CLI”
- MVP success criterion 1/2: checksum-verified CLI runs without runtime setup.

Why it matters:
The ADR chooses checksum verification, but does not define the trust root. If a binary and its checksum are both bundled in the same plugin payload, checksum verification only detects accidental corruption or mismatched packaging, not malicious tampering of the bundle. The risk register explicitly names “supply chain / tampered binary,” so the ADR needs to say what class of tampering checksum verification does and does not address.

Required fix:
Add a clear trust-boundary decision to ADR-002:
- where checksums live;
- whether they are generated from release CI;
- whether the checksum manifest is embedded in the selector, stored as a release manifest, signed, or simply bundled;
- what threat is mitigated: wrong-platform/mismatched/corrupt binary vs maliciously modified plugin package;
- whether signing/provenance is deferred.

If signing/provenance is deferred, record the residual risk explicitly in `knowledge/risks.md` instead of implying checksum verification fully resolves “tampered binary.”

Blocking finding 3 — High
Severity: High

Exact evidence:
- ADR-005 Context: “use atomic writes, avoid partial multi-file updates”
- ADR-005 Option A: “atomic write-and-rename, all-or-nothing multi-file staging under `.llm-wiki/`”
- ADR-005 Consequences: “interruption-safety (criteria 3, 20) follows from atomic+staged writes”
- ADR-005 Consequences: “cross-platform atomic-rename and staging semantics (notably Windows) need dedicated per-platform tests.”

Why it matters:
The ADR asserts all-or-nothing multi-file writes and interruption safety, but does not decide the transaction model. Atomic rename can protect one file, but it does not automatically make a multi-file operation atomic across platforms. The phrase “all-or-nothing multi-file staging” is doing too much work without defining the mechanism, especially because Windows semantics are explicitly called out as subtle.

Required fix:
ADR-005 needs either:
- a concrete high-level transaction model: staging directory, manifest, validation before commit, commit order, recovery/rollback behavior, temp-file cleanup, and how partial commits are detected; or
- a narrower claim: per-file atomicity is guaranteed in ADR-005, while cross-file transaction semantics are deferred to ADR-006 or another ADR.

Do not claim criteria 3/20 interruption safety “follows” unless the ADR actually defines enough mechanism to make that true.

Blocking finding 4 — Medium/High
Severity: Medium/High

Exact evidence:
- ADR-004 Decision: “Baseline mode (FR8) suppresses pre-existing findings on adoption while still flagging new or worsened ones.”
- Addendum 001 criterion 7: “Malformed YAML and missing profile-required fields fail validation.”
- ADR-004 Context: “invalid YAML / missing profile-required field / wrong field type = error”

Why it matters:
Baseline suppression can conflict with hard validation gates if it is allowed to suppress malformed YAML or required-field errors. Criterion 7 says malformed YAML and missing required fields fail validation. If a baseline can suppress these, the ADR creates ambiguity around whether validation is authoritative or merely differential during adoption.

Required fix:
Clarify baseline semantics:
- whether baseline can suppress errors at all;
- whether malformed YAML is non-baselineable because parsing must succeed before baseline comparison;
- whether missing required fields can be baseline-suppressed only in a special adoption mode that is not used for release-gate fixtures/CI;
- how exit codes behave when baseline suppresses findings.

This needs to be explicit before ADR-004 is accepted because it affects validation behavior, CI failure semantics, and acceptance criterion 7.

3. Non-blocking findings / refinements

- ADR-003 should eventually name the actual exit code numbers, not just the six semantic buckets. It is acceptable for the ADR to decide the set and leave numeric values to the implementation issue, but the acceptance hook “documented stable exit codes” will require exact values before Phase 1 closes.

- ADR-003 says human-readable output remains default for direct terminal use, while the shared JSON envelope is used by machine surfaces. That distinction is reasonable, but the ADR should explicitly say how direct CLI users request JSON, or whether all commands always have `--json`. Otherwise “one envelope shared by every surface” can be read as either mandatory default output or optional machine mode.

- ADR-004’s severity precedence is identified as hard but not decided: “core defaults → profile overrides → baseline suppression.” This is acceptable as a consequence only if Phase 1 issues are required to specify and test precedence before implementation. Consider adding one sentence to the Decision section, not just Consequences.

- ADR-004 introduces baseline mode, but the provided MVP/build-out excerpts do not foreground baseline as a major MVP acceptance concept. If it is required by PRD FR8, keep it. If not, consider whether it belongs in ADR-004 now or in a later validation/adoption ADR to avoid expanding Phase 1.

- ADR-005 says “treating inputs as data closes the untrusted-instruction risk.” That is directionally right, but it is not purely a filesystem-layer property. Skill prompting, adapter boundaries, and model/tool instruction handling also matter. Rephrase as “contributes to mitigation” unless another ADR owns the full instruction-injection boundary.

- Knowledge updates are mostly consistent: `knowledge/index.md` correctly says ADR-001–005 are proposed and pending human acceptance; `open-questions.md` keeps Q2/Q7/Q8 open with assumption locks; `risks.md` references the proposed ADRs without marking risks closed. That is the right status discipline.

- ADR-002 correctly defers install/upgrade/uninstall asset ownership to ADR-009. That boundary is good. Do not let Phase 2 implementation issues rely on ADR-002 alone for criteria 1, 3, or 20.

- Milestone sequencing is mostly sound: ADR-001/003/004/005 are valid Phase 1 prerequisites, ADR-002 is needed for OS/arch/checksum spine and Phase 2 packaging. ADR-006 remains a critical missing dependency for staged mutation, but it is correctly deferred because it does not block the engine-foundation ADR gate unless ADR-005 over-claims staged semantics.
