2. Status of prior blockers

Prior blocker 1 — ADR-006 rollback/preimage model underspecified
Status: fixed

- ADR-006 now states rollback is **preimage-backed, not hash-inferred**, and
  requires durable preimage records before the first mutating rename for every
  existing target that may be overwritten, deleted, or metadata-modified.
- Defines representations: absent targets via an explicit absent sentinel;
  regular files via preserved prior bytes plus ADR-005-managed metadata/mode bits.
- Defines rollback ordering: reverse commit order; atomic write+rename restore;
  rollback itself resumable from the journal. Roll-forward is available when
  staged postimages are intact. Both endpoints must be exact (fully committed
  tree, or byte-and-metadata-identical pre-commit tree).
- Assessment: the semantic gap is closed; the model no longer relies on hashes
  to reconstruct prior state. Implementable at ADR level.

Prior blocker 2 — ADR-009 ownership/version manifest underspecified
Status: fixed

- ADR-009 now specifies manifest path `.llm-wiki/manifest.json`; single JSON
  document; plugin-owned engine metadata; write only through the ADR-006
  transaction; removed by uninstall; pre-existing-non-manifest-file conflict
  behavior; minimum fields (`schemaVersion`, plugin/CLI/OKF versions, profile
  id+version, per-managed-asset `{ path, class, hash, lastInstalledHash }`);
  first-install, missing/unreadable/corrupt, and hash-inconsistent/user-modified
  behavior; default skip-and-report with explicit `--force` to overwrite/remove.
- Assessment: blocker resolved. Phase 2 implementers can place, protect, create,
  and validate the manifest; Phase 7 upgrade/uninstall policy is decided without
  forcing implementation into Phase 2.

3. Status of prior non-blocking findings

- ADR-007 phase split — fixed. Phase 3 owns the third-party registry/trust
  decision (or continued deferral); Phase 7 owns local-path custom-profile
  implementation; the profile format is fixed now, so later work changes only
  resolution, not the data model. Phase 2 stays limited to `core`.
- ADR-006 base-hash representation — fixed enough. Base hashes / preimage records
  over regular files only; absent sentinel distinct from empty-file hash;
  non-regular targets rejected at plan time; symlink/dir/device/socket handling
  aligned with ADR-005; directories created as needed but not transaction targets.
- ADR-009 installed/managed-asset wording — fixed. Distinguishes install-managed
  assets from arbitrary repo contents; plugin-owned (written, may be
  replaced/removed) vs repo-owned (recorded only where install must reference
  existing content); arbitrary pre-existing repo files are neither written nor
  catalogued.

4. New blocking findings

None.

5. Non-blocking findings (carry into implementation)

- ADR-006 leaves some latitude on roll-forward vs rollback selection; not
  blocking. Implementation must make the recovery decision deterministic,
  documented, and test-covered — especially under crash injection at each
  commit/rollback step.
- ADR-009 `hash` vs `lastInstalledHash` language could be sharper for repo-owned
  assets; not blocking. Implementation should define whether repo-owned records
  omit `lastInstalledHash`, set it null, or use it as "observed-at-install hash"
  (schema-level detail).
