2. Status of prior blockers

Prior blocker 1 — ADR-001 over-claimed comment preservation as binding
Status: resolved

Evidence:
- ADR-001 Context now says: “Acceptance criterion 6 ... is the binding requirement: unknown frontmatter fields survive authoring and enrichment” and separately: “Preserving YAML comments where feasible is a chosen design-quality bar, not a binding acceptance criterion.”
- ADR-001 Option A says comments are preserved “where feasible as a best-effort (non-gated) design-quality bonus.”
- ADR-001 Decision says: “comment preservation is a best-effort bonus, not the driver.”
- build-out-plan now says criterion 6 requires unknown-field round-trip preservation and “comment preservation is a best-effort, non-gated bonus.”
- knowledge/risks.md now says: “Acceptance criterion 6 (binding) requires unknown frontmatter field preservation; comment preservation is a best-effort, non-gated design-quality bar.”

Assessment:
This cleanly separates the binding acceptance criterion from the design-quality goal. No hidden MVP gate remains for comments.

Prior blocker 2 — ADR-002 lacked checksum trust-root / residual supply-chain risk language
Status: resolved

Evidence:
- ADR-002 Option A adds a Trust boundary section: checksums are “generated by release CI at build time and shipped as a manifest embedded in the plugin payload next to the binaries.”
- ADR-002 explicitly states this detects “wrong-platform, mismatched, truncated, or corrupted” binaries, but “not a malicious rewrite of the whole plugin/release payload.”
- ADR-002 Decision adds: “Checksums are CI-generated and bundled with the binaries, so verification is an integrity check ... not an authenticity check.”
- ADR-002 explicitly defers “Cryptographic signing / provenance attestation” to ADR-009 or a dedicated supply-chain ADR.
- knowledge/risks.md now records the residual risk: “the checksum manifest ships inside the same payload ... not a maliciously rebuilt payload ... signing / provenance attestation ... explicitly deferred ... risk stays open.”

Assessment:
The trust boundary is now explicit and no longer implies bundled checksums solve malicious supply-chain tampering.

Prior blocker 3 — ADR-005 over-claimed multi-file all-or-nothing interruption safety
Status: resolved

Evidence:
- ADR-005 Context now narrows scope: partial-write interruption is handled “at the single-file level” and “the multi-file transaction that makes a batch all-or-nothing is completed by ADR-006.”
- ADR-005 Option A says per-file atomic write+rename is “a necessary primitive” and explicitly says “per-file atomicity alone does not make a multi-file operation atomic.”
- ADR-005 Decision adds: “ADR-005 guarantees per-file atomicity only” and “does not define a cross-file transaction model.”
- ADR-005 lists the deferred transaction concerns: “staging directory layout, plan manifest, validate-before-commit ordering, commit sequencing, recovery/rollback, temp-file cleanup, and partial-commit detection.”
- ADR-005 Consequences say criteria 3 and 20 depend on this layer plus ADR-006, and “neither is fully satisfied by ADR-005 alone.”
- knowledge/risks.md mirrors this: ADR-005 gives the necessary primitive, while cross-file all-or-nothing transaction semantics are deferred to ADR-006.

Assessment:
The ADR no longer claims more than its mechanism supports. The ADR-005 / ADR-006 boundary is now credible.

Prior blocker 4 — ADR-004 baseline suppression conflicted with hard validation failures
Status: resolved

Evidence:
- ADR-004 Decision now defines severity precedence: “core defaults → profile overrides → baseline suppression.”
- ADR-004 says baseline suppression is “only as a differential filter, never as a severity change.”
- ADR-004 says: “Malformed YAML is never baseline-suppressible.”
- ADR-004 says missing profile-required fields are hard errors, and baseline suppression for them applies “only in the explicit adoption baseline mode” and is “not honored by release-gate / CI fixture runs.”
- ADR-004 exit-code behavior is explicit: if baseline suppresses the last remaining error the run exits success/success-with-warnings; if any unsuppressed error remains, including malformed YAML, it exits validation-failure.
- ADR-004 states: “CI / release-gate runs do not load an adoption baseline, so criterion 7 always fails the gate there.”

Assessment:
This resolves the ambiguity. Criterion 7 remains authoritative for CI/release gates, while adoption baseline mode is bounded and explicit.

4. Non-blocking findings / refinements

- ADR-003 still defers exact numeric exit-code values to the implementation issue. This is acceptable for ADR acceptance because the ADR fixes the semantic buckets and requires the numeric table before Phase 1 closes. Track it tightly in the Phase 1 issue.

- ADR-002 defers signing/provenance to “ADR-009 or a dedicated supply-chain ADR.” That is acceptable for this gate, but before packaging/release work begins the owner should decide whether ADR-009 actually owns it or whether a separate supply-chain ADR is needed. Otherwise the residual risk may stay open too long.

- ADR-005 Context still quotes the broad PRD phrase “avoid partial multi-file updates,” but the Decision and Consequences now clearly narrow ADR-005 to per-file atomicity and defer cross-file transaction semantics to ADR-006. Not blocking, but implementers should not read the Context sentence in isolation.

- knowledge/index.md still describes the current state as awaiting re-review, which is accurate before this review is accepted. After human acceptance, it should be updated to avoid stale process status.

- knowledge/log.md includes a later “2026-06-30” reconciliation entry after 2026-07-01 entries. That is chronological noise only, not an ADR gate issue.
